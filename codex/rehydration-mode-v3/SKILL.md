# Rehydration Mode V3 — 项目上下文再水化记忆系统（Codex 版）

## 核心理念

编程会话中断后重新开始时，最大痛点不是"不记得代码"，而是**不记得当时在做什么、做到哪了、为什么这样写**。

V3 的核心改进：
1. **Java 全栈检测** — 覆盖 Spring Boot/Cloud、Nacos、Kafka、Oracle 等主流框架和中间件
2. **SESSION_LOG 周文件封存** — 小时级条目 + 天级汇总 + 自然周归档
3. **PROJECT_MEMORY 三层记忆** — 热/温/冷分级管理 + 自动压缩建议

## 何时使用这个 Skill

1. **项目首次使用** — 初始化记忆系统
2. **会话启动恢复** — 继续上次工作
3. **会话结束保存** — 持久化当前状态
4. **会话中管理** — 更新进度、记录决策

## 能力 A：初始化再水化系统

### 触发条件
- 项目没有 `CODEX.md` 且没有 `docs/` 目录
- 用户明确说"初始化项目记忆"
- 用户说"帮我设置再水化系统"

### 执行步骤

**A1. 运行检测脚本**

```bash
python <skill_dir>/scripts/detect_project.py <project_root>
```

输出项目名、语言、框架、中间件、数据库等结构化 JSON。

**A2. 向用户确认**

```
📋 项目检测结果:
   名称: {project_name}
   技术栈: {inferred_stack}
   Git 仓库: {是/否}

我将创建以下记忆文件:

  ✅ CODEX.md                — 启动指令（Codex 每次进入自动加载）
  ✅ docs/PROJECT_MEMORY.md  — 项目整体记忆（热/温/冷三层结构）
  ✅ docs/CURRENT_TASK.md    — 当前任务状态（热恢复核心）
  ✅ docs/SESSION_LOG.md     — 会话日志（小时级 + 天级 + 周封存）
  ✅ docs/DECISIONS.md       — 架构决策记录（ADR）

确认以上信息？有什么需要补充或修改的？
```

**A3. 生成文件**

从 `assets/` 目录读取模板（`CODEX.md.tmpl`），替换占位符后写入项目。

- 技术栈自动填充（脚本检测到的内容直接填入）
- 保留模板注释（`<!-- -->` 注释是对用户的说明）
- 已存在文件先询问（不要静默覆盖）
- UTF-8 编码

**A4. 完成后提示**

```
✅ 再水化系统初始化完成！

现在可以：
  - 开始工作，过程中说"更新当前任务"保存进度
  - 会话结束时说"保存进度"持久化状态
  - 下次启动时说"继续上次的工作"即可恢复上下文

如果这是 Git 仓库，建议在 .gitignore 中添加：
  CODEX.local.md
```

## 能力 B：再水化（会话启动恢复）

### 触发条件
- 已初始化的项目，用户表达继续工作的意图
- 用户说"继续""接着做""上次做到哪了""恢复进度"

### 执行步骤

**B1. 检查文件结构**

```bash
python <skill_dir>/scripts/check_structure.py <project_root>
```

**B2. 检查 SESSION_LOG 是否需要跨周封存**

```bash
python <skill_dir>/scripts/session_log_manager.py <project_root> archive
```

**B3. 读取核心文件**（按顺序）

1. `docs/CURRENT_TASK.md` — **最先读**，记录上次编辑的文件、行号、阻塞项
2. `docs/PROJECT_MEMORY.md` 热记忆层 — 了解项目全貌
3. `docs/SESSION_LOG.md` 最新条目 — 了解最近的工作
4. `CODEX.md` — 了解项目规则和约束
5. `CODEX.local.md`（如果存在）— **只读不写**

**B4. 结构化汇报**

```
📋 {project_name} — 工作状态已恢复

🎯 阶段: {current_phase}（最后更新: {time}）

🔄 进行中 ({count} 项):
   {for each task:}
   ▸ {task_name} — {progress}%
     位置: `{file}:{line}`
     下一步: {next_step}

❌ 阻塞 ({count} 项):
   ▸ {blocker}（预计 {date} 解决）

💡 建议: {来自 CURRENT_TASK.md 的"下次会话建议"}
```

**原则**：
- 如果 CURRENT_TASK.md 中没有"进行中"任务，如实说"没有进行中的任务"
- 如果记忆文件存在但内容为模板默认值，告诉用户"记忆系统已初始化但尚未记录具体工作"
- 不要编造信息——所有内容必须来自已读取的文件

**B5. 确认方向**

```
是否从这个位置继续？或者你今天想做别的事？
```

## 能力 C：会话中的记忆管理

Codex 不支持 slash 命令，以下通过自然语句触发：

### "现在什么情况" — 快速概览

```
📊 记忆状态

项目: {name} | 阶段: {phase}
任务: {in_progress} 进行中 / {completed} 已完成 / {pending} 待开始
阻塞: {blocker_count} 项
最近会话: {从 SESSION_LOG.md 取最近 1-2 条}
最近决策: {从 DECISIONS.md 取最近 1-2 条}
记忆健康: {热记忆 X 行 / 温记忆 Y 行 / 冷记忆 Z 行}
```

### "更新当前任务" — 更新当前任务

1. 快速回顾 CURRENT_TASK.md 中的当前任务列表
2. 询问变更有哪些
3. 更新"最后更新"时间戳
4. 写入文件

**交互方式**（不要让用户填表格）：
```
当前记录的任务: {列出进行中的}
有什么变化？
  - 完成了 XX？我把它移到"已完成"
  - 新增了 YY？加到"进行中"
  - ZZ 遇到阻塞？加到"阻塞项"
```

### "记录日志" — 追加会话日志

- 自动记录：时间、模型
- 尝试 `git diff --stat` 获取代码变更
- 询问：本次完成了什么、有什么决策、遗留了什么
- 追加到 `docs/SESSION_LOG.md` **末尾**（最新在最下面）
- 如果是当天第一次会话，先创建 `## YYYY-MM-DD 日汇总` 标题

### "记录决策" — 记录架构决策

- 自动分配 ADR 编号（读 DECISIONS.md，取最大编号 +1）
- 日期自动填今天
- 状态默认"已决定 ✅"
- 只问必要的：背景、决策、原因、影响

### "整理记忆" — 记忆压缩

```bash
python <skill_dir>/scripts/memory_compressor.py <project_root> analyze
```

输出各层大小、占比、压缩建议。根据建议手动调整 PROJECT_MEMORY.md。

## 能力 D：会话结束脱水

### 触发条件
- 用户说"保存进度""结束工作""持久化""今天到这""脱水"

### 执行步骤

**D1. 更新 CURRENT_TASK.md**

- 已完成的任务从"进行中"移到"已完成"
- 仍在进行的任务保留，更新进度描述和当前编辑位置
- 新增的任务加入"待开始"
- 阻塞项状态同步
- "最后更新"时间戳刷新
- "下次会话建议"更新

**D2. 追加 SESSION_LOG.md**

追加到文件**末尾**（最新在最下面）：
- 小时级条目：`### {HH:MM} {会话标题}`
- 包含：完成的工作、关键决策、遇到的问题、遗留工作、代码变更
- 如果是当天最后一次会话，填写 `## YYYY-MM-DD 日汇总` 内容

**D3. 检查跨周封存**

```bash
python <skill_dir>/scripts/session_log_manager.py <project_root> archive
```

如果检测到跨周，自动将旧内容归档到 `session-log-YYYY-WXX.md`。

**D4. 检查架构决策**

本次会话中如果有重要决策，问用户是否记录到 DECISIONS.md。

**D5. 确认**

```
✅ 会话状态已保存

已更新:
  📝 CURRENT_TASK.md  — {N} 项任务进度已同步
  📋 SESSION_LOG.md   — 会话日志已追加
  {如有: 📐 DECISIONS.md — {N} 条新决策已记录}

下次启动时说"继续上次的工作"即可恢复。
```

## SESSION_LOG 文件结构（V3 新特性）

### 文件命名

```
docs/SESSION_LOG.md              ← 当前周活跃日志
docs/session-log-2026-W27.md     ← 第 27 周封存
docs/session-log-2026-W26.md     ← 第 26 周封存
```

### 小时级条目（每次会话追加）

```markdown
### {HH:MM} {会话标题}

**任务**: {目标}
**完成的工作**: ...
**关键决策**: ...
**遇到的问题**: ...
**遗留工作**: ...
**代码变更**: ...
```

### 天级汇总（每天首次/末次会话）

```markdown
## {YYYY-MM-DD} 日汇总

**今日完成**: ...
**关键决策**: ...
**遗留问题**: ...
**明日计划**: ...
```

### 周封存规则

- 跨周时自动归档到 `session-log-YYYY-WXX.md`
- 周文件只读，不再修改
- 当前文件重置为新周头部

详细格式规范见 `references/session_log_format.md`。

## PROJECT_MEMORY 三层记忆（V3 新特性）

### 结构

```
🔥 热记忆（Hot）   — 最近 7 天核心信息（架构/活跃模块/API/约定）
🌡️ 温记忆（Warm）  — 7-30 天次要信息（问题/环境/中间件/模块细节）
❄️ 冷记忆（Cold）  — 30 天+历史信息（废弃决策/旧API/历史环境）
```

### 压缩规则

- 热记忆 < 200 行，占比 < 60%
- 温记忆 < 100 行，14 天未引用 → 精简为摘要
- 冷记忆 < 50 行，保留标题 + 摘要指针

详细压缩规范见 `references/memory_compression.md`。

## 边界与安全

- **敏感信息**：绝不包含密钥、密码、Token
- **CODEX.local.md**：读取但不写入
- **授权边界**：只修改 `docs/` 目录下的记忆文件和 `CODEX.md`
- **覆盖保护**：生成文件前检查是否已存在
- **SESSION_LOG 封存**：每周自动归档，保留最近 3 个月周文件
