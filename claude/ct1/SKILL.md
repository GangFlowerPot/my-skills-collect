# ct1 — 创建多人 Agent 协作团队

## 何时使用

只要用户想搭建一个**有多角色分工、能并行工作、可查询进度**的 Agent 协作小组，就用这个 skill。典型触发：

- 中文：`创建团队`、`组建一个团队`、`搭建工作组`、`拉起一个小队`、`搞一个前后端测试组`、`成立专项小组`、`建个队伍`
- 英文：`create a team`、`build a team`、`set up a team`、`spin up a team`、`team up`
- 隐式意图：用户描述了一个需要多种角色协作完成的项目，即使没提"团队"二字

## 这个 skill 做什么

做三件事，按顺序：

1. **组建团队**：用 `Agent` 工具并行启动一组命名 Agent，每个都有明确角色、项目上下文和协作协议。
2. **建立协作规则**：让用户默认与"统筹领导决策者（leader）"对话；出方案时各技术角色分别出方案、测试出用例，由 leader 汇总；需求讨论时用户同时与 leader 和测试沟通（两个不同角度）。
3. **部署可复用进度查询机制**：生成一份协议文件并挂载到项目记忆导航，以后用触发词（`进度` / `status` / `progress` 等）随时拉取非 leader 成员的进度汇总表。

## 默认团队配置

| 角色 | Agent 名 | 模型 | 职责 | 与用户的关系 |
|---|---|---|---|---|
| 🎯 统筹领导决策者 | leader | 默认 Opus（跟随本会话） | 需求/方向沟通；汇总各方方案；方向决策；协调团队 | **默认对话对象** |
| 🖥️ 前端开发 | frontend-dev | 同上 | 前端实现方案（组件/状态/API/路由/交互） | 通过 leader |
| ⚙️ 后端开发 | backend-dev | 同上 | 后端实现方案（API/数据模型/服务/接口契约） | 通过 leader |
| 🧪 测试 | tester | 同上 | 前后端测试用例；需求阶段从质量/验收角度提问 | 需求讨论时与用户直接沟通 |

**leader 固定存在**，三个默认执行者可按用户需求替换、增减。

## 工作流（按顺序执行）

### 第 1 步：收集项目上下文

每个启动的 Agent 都需要知道它要参与的项目长什么样。按以下顺序探测，**取第一个能用的**：

1. 读项目根目录的 `CLAUDE.md` 或 `.claude/CLAUDE.md`（如果存在）
2. 读项目记忆导航入口（`AGENT_MEMORY.md` → `PROJECT_MEMORY.md`，如果存在）
3. 探测构建文件：`package.json`（前端技术栈/脚本）、`pom.xml` / `build.gradle`（后端技术栈/模块）、`go.mod`、`pyproject.toml` 等
4. 以上都没有 → 简短问用户："项目用什么技术栈？主要做什么？"（一两句话即可）

从上述来源提取一段**项目上下文摘要**（3–8 行即可），至少包含：项目名/领域、前端技术栈、后端技术栈、关键入口/命令。这段摘要会被注入到**每一个**启动的 Agent 的 prompt 里，所以不要写太长，但要让 Agent 能基于真实栈工作。

### 第 2 步：确认团队配置

先向用户展示默认配置（上表），然后用**一句话**问要不要调整：

> "默认是上面这四位（leader 固定 + 前端/后端/测试）。要调整吗？可以自然语言说，比如'加个 DBA'、'不要测试换两个后端'、'只要 leader 和一个全栈'。"

解析用户的修改意图，支持四类操作：**增**（加角色）、**删**（去角色）、**改**（改职责/技术侧重）、**换**（替换角色）。解析后生成一份**与默认配置的 diff**，展示给用户确认后再 spawn：

> "调整结果：去掉 tester，新增 devops（负责 CI/CD 与部署）。确认就开建？"

**校验规则**（解析后、确认前自动检查）：
- leader 必须存在且唯一（固定，不可删）
- 角色名/Agent 名不重复（大小写不敏感）
- 人数建议 2–8 人（leader + 1~7 执行者）；超过 8 人提醒并发与上下文成本
- Agent 名为字母数字 + 连字符，无空格；用户没给时根据其职责自动生成简洁英文名

**命名冲突处理**：如果同名 Agent 已在运行（如用户第二次"创建团队"），给用户三个选项——**重建**（停旧的重建，协议文件重置）、**复用**（不重建，只刷新当前任务）、**另建**（自动加 `-2`/`-3` 后缀，协议文件另存）。默认推荐"复用"。

**flat-roster 回退**：子 Agent 上下文（teammate）中，`Agent(name=...)` 可能被 harness 拒绝（"team roster is flat"）。遇到时按以下顺序回退，**不要中止流程**：
1. 去掉 `name` 参数，以 unnamed subagent 形式启动，把角色身份、项目上下文、协作协议、进度规范全部注入 prompt
2. 如果该角色在会话中已有活跃同名 teammate（命名冲突的"复用"路径），跳过启动，直接通过 `SendMessage` 向既有 teammate 发指令
3. 记录到 user_notes，继续后续步骤

回退后团队成员仍能正常工作（通过 SendMessage 用 agentId 或角色名寻址），只是不注册为命名 teammate。

### 第 3 步：并行启动 Agent

用 **一次消息里多个 `Agent` 调用**的方式并行启动所有角色，不要串行等待。每个 Agent 的 prompt 必须包含：

- **身份**："你是 [项目名] 的 [角色名]"
- **项目上下文摘要**：第 1 步提取的那段
- **技术栈约束**：基于探测到的真实栈，告诉它"必须基于现有栈，不引入项目未使用的框架"
- **协作协议**：本 skill 定义的沟通规则（默认对话对象、出方案规则、需求讨论规则）
- **进度查询规范**：当主线程发来 6 字段状态请求时，按格式回复（见"进度查询协议"节）
- **初始任务**：团队刚组建，确认就位，等待分配；**不要主动联系用户**（leader 除外，见下）

**model 字段**：默认用 `opus`（跟随本会话的 Opus 4.8）。如果用户在自定义时指定了其他模型，按用户指定。

**leader 特殊处理**：leader 的 prompt 还要加上"默认对话窗口"职责，并在末尾让它向用户做一个**简短团队介绍**（角色、工作模式、进度查询触发词），然后请用户提出第一个需求。

**命名冲突处理**：如果同名 Agent 已在运行（如用户第二次"创建团队"），在启动前问用户："已有同名团队，要替换（停掉旧的重建）还是改名新建？"

### 第 4 步：leader 介绍团队

leader 启动后会自动发介绍消息。主线程把它**原样转述**给用户，不要改写。等用户看到介绍后，团队就正式可用。

### 第 5 步：持久化进度查询协议

进度查询机制要跨会话可用，所以必须写进文件。生成 `references/team-protocol.md` 的**项目副本**：

- 如果项目有记忆目录（如 `skill-docs/`）→ 写到 `skill-docs/TEAM_PROTOCOL.md`
- 否则如果项目有 `.claude/` → 写到 `.claude/team-protocol.md`
- 否则 → 写到项目根目录 `team-protocol.md`

协议文件内容见 `references/team-protocol.md`（触发词、6 字段模板、汇总格式、执行规范）。部署时把"默认团队配置"替换成**本次实际组建的团队**（只列非 leader 成员）。

**验证副本（双写）**：除了上面的项目目录副本，再写一份到 `<skill目录>/ct1-workspace/team-protocol-snapshot.md`（全局 skill 工作区）。这一份不参与运行，仅用于：
- 离线验证"协议文件是否正确生成"而不污染项目
- 调试时对照项目副本是否被后续 eval/会话覆盖

两份内容必须一致（成员列表、触发词、6 字段模板）。

**并发写入注意**：如果目标项目文件已被其他 eval 迭代修改，Write 会触发"file has been modified since read"。处理方式：重新 Read → 确认当前内容 → 覆盖 Write。

然后**挂载导航**：
- 如果项目有 `AGENT_MEMORY.md` → 在其"记忆地图"表加一行指向该协议文件
- 否则如果项目有 `CLAUDE.md` → 在末尾加一段"团队协作"指向该文件
- 否则 → 在介绍中告知用户协议文件路径即可

## 进度查询协议

这是本 skill 部署的配套能力。协议文件是它的唯一真相源；下面摘要关键规则，细节以 `references/team-protocol.md` 为准。

### 触发词

用户说以下任一短语，主线程启动一次进度同步：

- 中文：`进度`、`查进度`、`进度如何`、`同步进度`
- 英文：`status`、`progress`、`check progress`、`where are we`、`how's it going`

### 查询流程

1. 识别触发词
2. **并行**向所有**非 leader** 执行者发送 6 字段状态请求（同一消息多个 `SendMessage`）
3. 收集回复，汇总为一张 Markdown 表格
4. 未回复者显示 `⏳ 未响应`

### 6 字段状态请求模板（发给每位执行者）

```
请按以下格式报告你当前的状态（简洁即可）：
【状态】空闲 / 工作中 / 阻塞
【当前任务】一句话描述
【进度】百分比或关键里程碑
【阻塞项】无 / 具体描述
【下一步】即将做什么
【需要的输入】无 / 需要什么（如需求、接口契约、配置）
```

### 汇总表格格式

```
| 成员 | 状态 | 当前任务 | 进度 | 阻塞项 | 下一步 | 需要的输入 |
|---|---|---|---|---|---|---|
| 🖥️ 前端 | ... | ... | ... | ... | ... | ... |
| ⚙️ 后端 | ... | ... | ... | ... | ... | ... |
| 🧪 测试 | ... | ... | ... | ... | ... | ... |
```

表格列随本次实际团队角色变化。

## 协作规则摘要（注入每个 Agent）

- **默认对话**：用户 ↔ leader
- **出方案**：各技术角色各自出方案，测试同步出覆盖全栈的测试用例 → leader 汇总后呈报用户
- **沟通需求**：用户同时与 leader（产品/风险/进度/资源角度）和测试（质量/边界/用户体验/验收标准角度）对话
- **进度查询**：主线程按触发词并行查非 leader 成员，不经过 leader

## 注意事项

- **并行启动**：第 3 步必须一次消息多 Agent，避免串行等待拖慢组建
- **上下文摘要要短**：每个 Agent 都注入，3–8 行足够；细节让 Agent 按需 Read
- **凭据不入 prompt**：项目上下文摘要里不要包含密码、Token、私钥等秘密
- **技术栈跟着项目走**：默认角色名是"前端/后端/测试"，但 prompt 里的技术约束必须来自第 1 步探测到的真实栈，不要硬编码 Vue/Java
- **leader 是枢纽但进度查询绕过它**：进度查询直接查执行者，不增加 leader 负担
