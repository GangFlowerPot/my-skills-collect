# 上下文合约（Context Contract）

## 是什么

上下文合约是一份**项目级、可复用**的角色→文档切片映射。它声明每个角色需要哪些结构化文档的哪些部分，让子 agent 的 spawn prompt 能按角色自动组装，而不是每次手写或统一注入同一段摘要。

## 为什么需要

当前 ct1 把同一段 3–8 行技术栈摘要注入**每一个** agent（`SKILL.md` 第 1 步 / 第 3 步），导致四个问题：

| 痛点 | 合约如何解决 |
|---|---|
| **信息衰减**：leader 转述后失真 | 合约指向文档源头，agent 直读原始切片 |
| **Token 浪费**：20KB 全文塞入 | 按角色切片，~5KB 精准注入 |
| **对齐成本高**：开放任务格式乱 | 配合五要素模板 + 格式锚点（见 `five-element-prompt.md`） |
| **编写瓶颈**：每次手写 prompt | 合约复用，主线程只填任务字段 |

## Schema

一份上下文合约包含：**文档清单** + **每个角色的切片表** + **按需引用列表**。

```markdown
# 上下文合约 — [项目名]

## 文档清单
项目结构化文档的全貌（合约的"货源"）：

| 文档 | 路径 | 总大小 | 说明 |
|---|---|---|---|
| 需求文档 | docs/req.md | 20KB | 功能/交互/验收/边界 |
| 架构文档 | docs/architecture.md | 15KB | 前后端架构/数据模型/API 契约 |
| 决策记录 | docs/decisions.md | 8KB | ADR 列表 |

## 角色切片

### 🖥️ 前端开发（frontend-dev）
**目标：~5KB**

| 文档 | 切片范围 | 大小 | 用途 |
|---|---|---|---|
| req.md | §3.1–3.5 页面与交互需求 | 2KB | 知道要做什么 |
| req.md | §5 验收标准 | 1KB | 知道什么样算做完 |
| architecture.md | §2 前端架构 + §4 API 契约 | 1.5KB | 知道技术边界 |
| decisions.md | §D1, D3（UI 相关决策） | 0.5KB | 知道已经定的事 |

### ⚙️ 后端开发（backend-dev）
**目标：~5KB**

| 文档 | 切片范围 | 大小 | 用途 |
|---|---|---|---|
| req.md | §4 业务逻辑 | 2KB | 知道要做什么 |
| req.md | §5 验收标准 | 1KB | 知道什么样算做完 |
| architecture.md | §3 后端架构 + §5 数据模型 | 1.5KB | 知道技术边界 |
| decisions.md | §D2, D4（数据/接口决策） | 0.5KB | 知道已经定的事 |

### 🧪 测试（tester）
**目标：~4KB**

| 文档 | 切片范围 | 大小 | 用途 |
|---|---|---|---|
| req.md | §5 验收标准 + §6 边界与异常 | 2KB | 知道验什么 |
| architecture.md | §6 集成点与依赖 | 1.5KB | 知道哪里容易出问题 |
| decisions.md | 全部（质量相关） | 0.5KB | 知道质量决策 |

## 按需引用（不嵌入，agent 自读）
- req.md §6 异常流程详情 → 测试遇到边界问题时自读
- architecture.md §7 部署拓扑 → devops 角色按需自读
```

## 字段说明

| 字段 | 含义 |
|---|---|
| 文档 | 结构化文档名（req.md / architecture.md / decisions.md 等） |
| 切片范围 | 章节号或标题，能唯一锚定一段内容 |
| 大小 | 预估 token/字符量，用于控制单角色注入总量 |
| 用途 | 为什么这个切片对该角色必要（防止过度注入） |

**大小预算**：单角色嵌入总量建议 **3–8KB**。超过 8KB 说明切片过宽，应拆成"嵌入 + 按需引用"。

## 如何建立

1. **盘点项目文档**：列出所有结构化需求/架构/决策文档（req.md、architecture.md、decisions.md 等）
2. **定义角色列表**：基于本次团队配置（见 `team-protocol.md`）
3. **逐角色填切片**：问"该角色最少需要知道什么才能开始工作"，填切片范围；控制单角色 ~5KB
4. **标注按需引用**：非核心但可能有用的文档位置，留给 agent 自读
5. **存储**：写入 `zsh/CONTEXT_CONTRACT.md` 或 `TEAM_PROTOCOL.md` 的上下文合约章节

## 维护规则

- **文档更新时**：检查合约中的切片范围是否仍然有效（章节是否重排、内容是否迁移），必要时调整
- **新增角色时**：补一段切片表
- **任务跨度大时**（如从前端扩展到全栈）：重新评估切片范围，可能需要替换而非追加
- **时效验证**（每次 spawn 前）：对比合约中文档的 `最后更新` 时间戳与实际文件 mtime，若文件已更新但合约未同步，触发重新切片

## 与五要素模板、动态补充协议的关系

- **上下文合约** = 静态的"货源地图"（角色需要什么）
- **五要素模板**（`five-element-prompt.md`）= 动态的"组装配方"（怎么拼 prompt）
- **动态补充协议**（`dynamic-supplement-protocol.md`）= 运行时的"补给通道"（任务推进中怎么补信息）

三者配合：合约提供切片 → 模板组装成 prompt → 补充协议在运行时增量更新。
