# 动态补充协议（Dynamic Supplement Protocol）

## 是什么

任务推进中，leader / 主线程向子 agent 灌入新信息的**结构化协议**，替代 leader 的自由格式总结，避免信息衰减。

## 为什么需要

任务推进中常出现新信息：
- leader 做了决策（技术选型、需求裁剪）
- 跨 agent 输出变化（后端接口变更 → 前端需要知道）
- 用户追加需求

如果 leader 用自由格式转述，会引入信息衰减。结构化补充保持信息保真度。

## 补充消息 schema

```
[CONTEXT ADDENDUM]
来源：leader 决策 / 用户新需求 / 跨 agent 输出（如后端接口变更）
关联任务：[任务标识]
内容：[具体信息，直接可操作]
对原有上下文的变更：[新增 / 替换 §X / 作废 §Y]
```

### 字段说明

| 字段 | 说明 |
|---|---|
| 来源 | 信息出处，便于 agent 评估权威性 |
| 关联任务 | 这条补充对应哪个任务/问题 |
| 内容 | 直接可操作的具体信息（不要总结，给原始决策/数据） |
| 对原有上下文的变更 | 这条信息与已注入切片的关系：新增、替换某切片、作废某切片 |

## 两种补充通道

### 推送（leader 主动发）
leader 在有新信息时，按 schema 发送 `[CONTEXT ADDENDUM]` 给相关 agent。

示例：
```
[CONTEXT ADDENDUM]
来源：leader 决策 — 与用户讨论后确定技术选型
关联任务：前端组件拆分方案（Q2 状态管理选型）
内容：列表页状态管理统一用 Pinia，不用组件本地状态。原因：列表数据需在多个组件间共享。
对原有上下文的变更：替换 decisions.md §D1（原"待定" → 现"Pinia"）
```

### 拉取（agent 自服务）
agent 遇到信息不足时，按 `context-contract.md` 合约中的「按需引用」自行 Read 更深层文档。

规则：
- 先判断缺失信息是否已在合约「按需引用」中 → 是则直接 Read
- 如果缺失信息超出合约范围（全新领域） → 向 leader 请求 `[CONTEXT ADDENDUM]`，说明需要什么

## 与合约的关系

动态补充是合约的**运行时扩展**：
- 合约为**静态基线**（spawn 时一次性注入）
- 补充为**动态增量**（任务推进中按需叠加）

变更类型决定如何更新合约：
- **新增**：临时追加，不修改合约（合约下次维护时再评估是否纳入基线）
- **替换 §X**：持久变更，应同步更新合约中的切片内容（保持合约时效性）
- **作废 §Y**：持久变更，应从合约中移除对应切片

## 完整流程

```
1. spawn：按合约注入静态切片简报（~5KB）
2. 任务执行：agent 基于简报工作
3. 出现新信息：
   a. leader 发送 [CONTEXT ADDENDUM]（推送）
   b. 或 agent 按合约自读（拉取）
4. agent 按「对原有上下文的变更」字段更新本地理解
5. 继续执行任务，输出
```

## 注意事项

- **补充消息也要精简**：一条补充聚焦一个变更，不要打包多个不相关信息
- **保持 schema**：即使来源简单，也写明「来源 / 关联任务 / 内容 / 变更」，便于 agent 定位
- **避免循环补充**：如果 agent 反复请求同一类补充，说明合约切片不足，应回头补合约而非持续打补丁
