# 团队进度查询协议（模板）

本文件是进度查询机制的唯一真相源。由 `ct1` skill 部署到项目，部署时把"默认团队配置"替换为实际组建的非 leader 成员。

## 触发词

用户发送以下任一短语，主线程启动一次**进度同步**：

- 中文：`进度` / `查进度` / `进度如何` / `同步进度`
- 英文：`status` / `progress` / `check progress` / `where are we` / `how's it going`

## 查询流程（主线程执行）

1. 识别触发词
2. **并行**向以下非 leader 成员发送「状态请求模板」（同一消息多个 `SendMessage`，避免串行）
3. 收集回复，按「汇总表格格式」整理为 Markdown 表格
4. 超时未回复者对应行显示 `⏳ 未响应`，表后提示"可再次发送触发词重新查询"

## 默认团队（部署时替换为实际成员）

进度查询的成员范围（不含 leader）：

- 🖥️ 前端（frontend-dev）
- ⚙️ 后端（backend-dev）
- 🧪 测试（tester）

## 状态请求模板（发给每位执行者）

```
请按以下格式报告你当前的状态（简洁即可）：
【状态】空闲 / 工作中 / 阻塞
【当前任务】一句话描述
【进度】百分比或关键里程碑
【阻塞项】无 / 具体描述
【下一步】即将做什么
【需要的输入】无 / 需要什么（如需求、接口契约、配置）
```

## 汇总表格格式

```
| 成员 | 状态 | 当前任务 | 进度 | 阻塞项 | 下一步 | 需要的输入 |
|---|---|---|---|---|---|---|
| 🖥️ 前端 | ... | ... | ... | ... | ... | ... |
| ⚙️ 后端 | ... | ... | ... | ... | ... | ... |
| 🧪 测试 | ... | ... | ... | ... | ... | ... |
```

> 列和行随实际团队角色调整。成员列用"emoji + 角色名"标识。

## 执行者回复规范

执行者收到状态请求后，**严格按 6 字段、固定顺序**回复，不额外寒暄，便于主线程解析。示例：

```
【状态】空闲
【当前任务】等待分配需求
【进度】—
【阻塞项】无
【下一步】待 leader 派发任务
【需要的输入】首个需求描述
```

## 协议复用说明

本文件是进度查询机制的唯一真相源。主线程识别到触发词后，按本文件规定的模板和流程执行，不依赖临时记忆。会话重建后，通过项目记忆导航 → 本文件恢复机制。
