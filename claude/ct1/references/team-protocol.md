# 团队进度查询协议（模板）

本文件是进度查询机制的唯一真相源。由 `ct1` skill 部署到项目，部署时把"默认团队配置"替换为实际组建的非 leader 成员。

## 触发词

用户发送以下任一短语，主线程启动一次**进度同步**：

- 中文：`进度` / `查进度` / `进度如何` / `同步进度`
- 英文：`status` / `progress` / `check progress` / `where are we` / `how's it going`

## 查询流程（主线程执行）

1. 识别触发词
2. **并行**向以下非 leader 成员发送「状态请求模板」（同一消息多个 `SendMessage`，避免串行）
3. 收集回复，按「汇总表格格式」整理为 Markdown 表格
4. 超时未回复者对应行显示 `⏳ 未响应`，表后提示"可再次发送触发词重新查询"

## 默认团队（部署时替换为实际成员）

进度查询的成员范围（不含 leader）：

- 🖥️ 前端（frontend-dev）
- ⚙️ 后端（backend-dev）
- 🧪 测试（tester）
- 🔍 审查（reviewer）

## 状态请求模板（发给每位执行者）

```
请按以下格式报告你当前的状态（简洁即可）：
【状态】空闲 / 工作中 / 阻塞
【当前任务】一句话描述
【进度】百分比或关键里程碑
【阻塞项】无 / 具体描述
【下一步】即将做什么
【需要的输入】无 / 需要什么（如需求、接口契约、配置）
【最近上报里程碑】33% / 66% / 100% / 无
【待答复问题】无 / 当前卡点的问题（详见「问题升级循环」节）
【本轮完成文件】无 / 本轮新增或修改的文件列表（里程碑节点填写，供审查）
```

**后三个字段说明**：
- `【最近上报里程碑】`：本次回复中问题所属的里程碑（33% / 66% / 100%），非里程碑节点填「无」
- `【待答复问题】`：仅在有疑问需用户决策时填写（格式见「问题升级循环」节），无疑问填「无」
- `【本轮完成文件】`：里程碑节点填写本轮新增/修改的文件路径列表（供 leader 送 reviewer 审查），非里程碑或无文件变更填「无」

问题升级循环的完整规范（子 agent 问题记录、里程碑检查点、leader 聚合与答复分发）见 `references/question-escalation-protocol.md`。审查循环的完整规范（reviewer 角色、审查报告、三轮循环）见 `references/code-review-protocol.md`。

## 汇总表格格式

```
| 成员 | 状态 | 当前任务 | 进度 | 阻塞项 | 下一步 | 需要的输入 | 最近上报里程碑 |
|---|---|---|---|---|---|---|---|
| 🖥️ 前端 | ... | ... | ... | ... | ... | ... | ... |
| ⚙️ 后端 | ... | ... | ... | ... | ... | ... | ... |
| 🧪 测试 | ... | ... | ... | ... | ... | ... | ... |
| 🔍 审查 | ... | ... | ... | ... | ... | ... | ... |
```

> 列和行随实际团队角色调整。成员列用"emoji + 角色名"标识。「最近上报里程碑」为固定新增列。完整的双 section 展示格式（含待答复问题）见「leader 聚合展示格式」节。reviewer 的状态为 空闲/审查中/完成。

## 执行者回复规范

执行者收到状态请求后，**严格按 8 字段、固定顺序**回复，不额外寒暄，便于主线程解析。

### 无疑问时的示例

```
【状态】空闲
【当前任务】等待分配需求
【进度】—
【阻塞项】无
【下一步】待 leader 派发任务
【需要的输入】首个需求描述
【最近上报里程碑】无
【待答复问题】无
```

### 在 33% 里程碑遇到问题时的示例

```
【状态】工作中
【当前任务】前端登录模块开发
【进度】33%（完成页面骨架与状态管理）
【阻塞项】无
【下一步】等待答复后决定登录态实现方案，继续实现登录表单
【需要的输入】无
【最近上报里程碑】33%
【待答复问题】
1. [Q-FE-001][33%] 是否采用方案A（JWT+refresh token）或方案B（session）实现登录态？ (关联任务: 前端登录模块)
2. [Q-FE-002][33%] 表单验证错误提示文案是否沿用旧版？ (关联任务: 前端登录模块)
【本轮完成文件】
- src/components/LoginForm.vue（新增）
- src/store/user.js（修改）
```

**问题 ID 格式**：`Q-{role}-{seq}`（如 `Q-FE-001`、`Q-BE-001`），seq 按 agent 自增。

### 里程碑检查点行为

执行者在每次进度更新时执行里程碑检查点（详见 `references/question-escalation-protocol.md`）：
- 进度抵达 33% / 66% / 100% 时，把所有「待上报」问题标记为「已上报」并填入【待答复问题】
- 跳过里程碑（如 30%→70%）时，依次补报 33% 和 66% 检查点的积压问题
- **不暂停工作**：遇到问题先跳过该部分继续推进其他工作，等答复到达后回填

### reviewer 状态示例

reviewer 不参与编码，其状态为 空闲/审查中/完成：

```
【状态】审查中
【当前任务】审查 frontend-dev 的运单列表页代码（第 1 轮 / 33%）
【进度】—
【阻塞项】无
【下一步】输出 Code Review 报告
【需要的输入】无
【最近上报里程碑】无
【待答复问题】无
【本轮完成文件】无
```

审查循环的完整规范（reviewer 角色定义、审查报告 schema、leader 分流、三轮循环）见 `references/code-review-protocol.md`。

## leader 聚合展示格式

主线程收集回复后，**同一条消息**内分两个 section 展示给用户：

```
📊 团队进度报告 · {时间}

## 一、进度总览
| 角色 | 状态 | 当前任务 | 进度 | 阻塞项 | 下一步 | 需要的输入 | 最近上报里程碑 |
|------|------|----------|------|--------|--------|------------|----------------|
| 🖥️ 前端 | ... | ... | ... | ... | ... | ... | ... |
| ⚙️ 后端 | ... | ... | ... | ... | ... | ... | ... |
| 🧪 测试 | ... | ... | ... | ... | ... | ... | ... |

## 二、待答复问题（共 N 题）
| 编号 | 来源 | 里程碑 | 问题 | 关联任务 | 提出次数 |
|------|------|--------|------|----------|----------|
| Q-FE-001 | frontend-dev | 33% | ... | ... | 1 |
| Q-BE-001 | backend-dev | 33% | ... | ... | 1 |

请回答上述问题。可部分回答，格式如「Q-FE-001: 方案A」；跳过的题目将保留至下次询问。
```

**展示规则**：
- 表格列随实际团队角色变化；新增固定的「最近上报里程碑」列
- 「待答复问题」section 仅当任一成员有非「无」的问题时出现，否则不显示
- leader 对跨 agent 的相似问题做语义去重（合并来源），重复问题通知来源 agent 作废
- 用户回答后，leader 通过 `[CONTEXT ADDENDUM]`（来源=用户答复）分发给对应 agent（详见 `references/question-escalation-protocol.md`）

## 协议复用说明

本文件是进度查询机制的唯一真相源。主线程识别到触发词后，按本文件规定的模板和流程执行，不依赖临时记忆。会话重建后，通过项目记忆导航 → 本文件恢复机制。
