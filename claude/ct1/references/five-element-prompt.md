# 五要素 Prompt 模板

## 是什么

子 agent spawn prompt 的标准化结构。五要素齐全 = 自包含，agent 无需追问就能开始工作。

```
[角色]         ← 你是谁
[上下文]       ← 按合约切出的角色简报（~5KB）
[任务]         ← 具体化任务（问题/交付物）
[文档引用]     ← 可进一步 Read 的位置（按需自读）
[输出格式]     ← 明确的输出格式模板
```

## 各要素写法

### 1. 角色
```
你是 [项目名] 的 [角色名]，负责 [一句话职责]。
```

### 2. 上下文
直接粘贴按 `context-contract.md` 合约切出的角色简报（~5KB）。这是 agent 工作所需的最小知识集，**按角色不同**——前端拿前端切片，后端拿后端切片。

### 3. 任务（必须具体化）

**反模式**（避免）：
- ❌ "输出你的疑问"（开放任务，格式不可控）
- ❌ "分析这个需求"（粒度不清）
- ❌ "做一下前端方案"（范围模糊）

**正模式**（具体化）：
```
任务：根据上述上下文，输出前端组件拆分方案：
  Q1. 需要哪些页面/组件？列出组件树（不超过 3 层）
  Q2. 每个组件的 props 和状态是什么？
  Q3. 哪些现有组件/逻辑可复用？给出文件路径
```

任务具体化三问：
- 交付物是什么？（方案 / 列表 / 代码）
- 交付几项？（明确的项数）
- 每项的边界是什么？（不超过 X 层 / 不超过 Y 个）

### 4. 文档引用
```
如需了解更多，可自读：
- req.md §6 异常流程详情
- architecture.md §7 部署拓扑
```
只列位置，不嵌入内容。agent 遇到信息不足时按需 Read。

### 5. 输出格式（格式锚点）
给 agent 一个填空式模板，减少决策负担：

```
请按以下格式输出：
## 组件树
（树形结构，不超过 3 层）
## 组件明细
| 组件名 | props | 状态 | 复用来源（文件路径） |
|--------|-------|------|----------------------|
|        |       |      |                      |
## 风险与假设
- （列出做方案时的前提假设和潜在风险）
```

## 完整示例

```
你是 ynwl 物流平台的前端开发，负责前端实现方案（组件/状态/API/路由/交互）。

## 上下文
（此处粘贴按合约切出的前端简报，~5KB，包含 req.md §3.1-3.5 + §5 + architecture.md §2+§4 + decisions.md §D1,D3）

## 任务
根据上述上下文，输出列表页的前端组件拆分方案：
  Q1. 需要哪些组件？列出组件树（不超过 3 层）
  Q2. 每个组件的 props 和状态是什么？
  Q3. 哪些现有组件可复用？给出文件路径

## 文档引用
如需了解更多，可自读：
- req.md §6 异常流程详情
- architecture.md §7 部署拓扑（了解接口部署约束）

## 输出格式
请按以下格式输出：
## 组件树
...
## 组件明细
| 组件名 | props | 状态 | 复用来源 |
...
## 风险与假设
...
```

## 与合约的关系

五要素模板的「上下文」字段由 `context-contract.md` 合约自动填充。主线程 spawn 时：
1. 读合约，找到该角色的切片清单
2. 从项目文档中切出对应片段
3. 嵌入 prompt 的「上下文」字段
4. 把合约中的「按需引用」部分填入「文档引用」字段

→ 主线程只需编写「任务」+「输出格式」，上下文由合约自动组装。
