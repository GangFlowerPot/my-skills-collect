# 端到端测试：ynwl 项目的上下文灌输全链路演示

> 这是一个**具体的工作示例**，用真实项目（ynwl 物流管理系统）演示上下文灌输机制的完整链路：
> 合约建立 → 按角色切片 → 五要素 prompt 组装 → token 预算 → 动态补充。
>
> 测试目标（来自方案「验证方式」）：
> 1. 信息保真：子 agent 读的是文档源头切片，而非 leader 总结
> 2. Token 效率：单角色 ~5KB，而非 20KB 全文
> 3. 对齐成本：格式锚点让首次输出直接命中模板
> 4. 编写成本：主线程只填「任务 + 输出格式」，上下文由合约自动组装
> 5. 动态补充：`[CONTEXT ADDENDUM]` 能被 agent 接收并更新行为

---

- 项目 ynwl：前后端分离的管理系统

- 微服务：blade-gateway(10100)、blade-auth、blade-system、blade-user、blade-fileserver、blade-apires、blade-log
- 注册中心 Nacos

## 2. 主线程编写量对比 — 验证「编写成本」

### 旧机制（当前 ct1）：主线程为每个 agent 手写
主线程需要为 3 个执行者分别编写完整 prompt（身份 + 相同的 3-8 行摘要 + 技术约束 + 协作协议 + 进度规范 + 初始任务）。每次 spawn 都要重写整段。

### 新机制：主线程只填「任务 + 输出格式」
上下文由合约自动组装。主线程每次 spawn 只需编写以下**任务特定部分**（其余由合约 + 模板复用）：

```

  Q3. 哪些现有组件可复用？给出文件路径

## 输出格式
请按以下格式输出：

（树形结构，不超过 3 层）
## 组件明细
| 组件名 | props | 状态 | 复用来源（文件路径） |
|--------|-------|------|----------------------|
|        |       |      |                      |
## 风险与假设
- （列出做方案时的前提假设和潜在风险）
```

**编写成本估算**：
- 旧机制：每个 agent 主线程手写 ~200 字（含上下文摘要），3 个 agent = ~600 字
- 新机制：主线程只写「任务 + 格式」~100 字，3 个 agent 共用同一份（仅角色名不同）= ~100 字
- **降幅 ~83%**（远低于「< 原来 30%」目标）

## 5. 动态补充演示 — 验证「动态补充协议」

**场景**：任务推进中，后端决定把运单列表接口从同步改为异步（大数据量时返回任务 ID，轮询结果）。前端需要知道这个变更。

**leader 发送**：
```
[CONTEXT ADDENDUM]
来源：backend-dev 输出变更 — 运单列表接口改为异步模式
关联任务：前端组件拆分方案（Q2 状态管理 + API 对接）
内容：运单列表接口 GET /api/waybill/list 在大数据量（> 1000 条）时改为异步：
  - 第一次调用返回 { taskId }，前端轮询 GET /api/waybill/task/{taskId}
  - 轮询间隔 2s，超时 30s 显示"查询超时，请缩小筛选范围"
  - 状态码：PROCESSING / SUCCESS / FAILED
  - 前端需在 waybillStore 中新增 polling 状态管理
对原有上下文的变更：替换 architecture.md §5 API 契约（同步 → 异步）+ 新增轮询交互需求
```

**前端 agent 接收后**：
1. 按「对原有上下文的变更」字段，更新本地对 API 契约的理解（同步 → 异步）
2. 在组件方案中补充轮询状态管理（waybillStore.polling）
3. 在「风险与假设」中新增一条：异步轮询对用户体验的影响（需 loading 提示）
4. **不需要**重新读全文，只需基于增量更新

**验证点**：
- 补充消息结构化（来源 / 关联任务 / 内容 / 变更）→ agent 能精确定位影响范围
- 「替换 §X」标记 → 下次合约维护时，应同步更新合约中的 API 契约切片（保持合约时效性）
- 没有 leader 的自由总结 → 信息保真
