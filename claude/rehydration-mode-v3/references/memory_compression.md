# PROJECT_MEMORY.md 记忆压缩规范（V3）

## 三层记忆模型

```
🔥 热记忆（Hot）   — 最近 7 天频繁使用的核心信息
🌡️ 温记忆（Warm）  — 7-30 天内使用过的次要信息
❄️ 冷记忆（Cold）  — 超过 30 天或极少使用的历史信息
```

## 各层内容指南

### 🔥 热记忆（Hot）

**保留内容**：
- 项目概述（2-3 句话）
- 架构概览（Mermaid 图）
- 当前活跃模块表（仅正在开发的模块）
- 核心 API（最近 7 天使用过的）
- 关键约定（命名、代码组织、分支策略）

**大小限制**：不超过总文件的 60%，建议 < 200 行

**更新频率**：每次架构变更时更新

### 🌡️ 温记忆（Warm）

**保留内容**：
- 数据模型（实体关系图、表结构）
- 已知问题 / 技术债务（按优先级排列）
- 环境信息（开发/测试/生产地址）
- 中间件配置（注册中心、消息队列、缓存等）
- 模块细节（非当前活跃模块的说明）

**大小限制**：建议 < 100 行

**更新频率**：每周 review 一次

**压缩规则**：
- 超过 14 天未引用的条目精简为 1-2 行摘要
- 已解决的问题移到冷记忆
- 环境信息变更后旧信息移到冷记忆

### ❄️ 冷记忆（Cold）

**保留内容**：
- 已废弃决策（标题 + 指向 DECISIONS.md 的指针）
- 历史环境（已下线的环境说明）
- 历史 API（已废弃的接口）
- 已完成的里程碑（仅保留标题和日期）

**大小限制**：建议 < 50 行

**更新频率**：每月归档一次

**压缩规则**：
- 详细内容替换为摘要指针
- 指向 SESSION_LOG 中的原始记录
- 指向 DECISIONS.md 中的 ADR 记录

## 压缩执行流程

### 1. 分析当前状态

```bash
python <skill_dir>/scripts/memory_compressor.py <project_root> analyze
```

输出各层大小、占比、建议操作。

### 2. 获取压缩建议

```bash
python <skill_dir>/scripts/memory_compressor.py <project_root> compress
```

输出具体的压缩建议（dry run 模式，不实际修改文件）。

### 3. 手动执行压缩

根据建议，手动调整 PROJECT_MEMORY.md 的内容：
- 将过旧的温记忆条目精简为摘要
- 将极少使用的内容降级到冷记忆
- 确保热记忆占比不超过 60%

### 4. 快速统计

```bash
python <skill_dir>/scripts/memory_compressor.py <project_root> stats
```

## 压缩示例

### 压缩前（温记忆中的详细条目）

```markdown
### 已知问题 / 技术债务

1. **数据库连接池配置不合理**
   - 问题: Druid 连接池 maxActive 设置为 500，实际并发只有 50，浪费资源
   - 影响: 高并发时可能出现连接耗尽
   - 计划: v0.6 优化连接池配置
   - 优先级: P2
   - 更新: 2026-06-15
```

### 压缩后（精简为摘要）

```markdown
### 已知问题 / 技术债务

- **Druid 连接池配置** — maxActive=500 过高，计划 v0.6 优化（详见 SESSION_LOG 2026-W24）
```

### 冷记忆中的归档格式

```markdown
### 已完成的里程碑

- [x] v0.1 项目脚手架搭建 (2026-06-01)
- [x] v0.2 用户认证模块 (2026-06-10)
- [x] v0.3 订单基础 CRUD (2026-06-20)
```
